### Architecture & Patterns
- **Orchestration Layer**: The project uses **Microsoft AutoGen (0.4+)** to coordinate multiple LLMs into an interactive team environment. Models operate as `AssistantAgent` instances mapped through either a `RoundRobinGroupChat` (Broadcast mode) or `SelectorGroupChat` (Discuss mode).
- **Presentation Layer**: Built on top of **Chainlit**. The UI is heavily customized using CSS (`public/style.css`) to suppress modern UI elements (like avatars, rounded corners, padding) and enforce a strict 1990s IRC terminal aesthetic (monospaced fonts, dark backgrounds, `<Nickname>` formatting).
- **Separation of Concerns**:
  - `app.py`: Acts as the presentation and routing layer. It binds Chainlit's async event loops (`@cl.on_chat_start`, `@cl.on_message`), handles chat settings and command parsing (`/slap`, `/mode`, `/topic`), processes multi-modal inputs (like vision images), and manages the streaming of AutoGen responses.
  - `simulator_core.py`: Contains the core business logic, session state definition, transcript formatting, UI string generation, and data persistence operations for preserving state to `data/simulator_state.json`.
  - `simulator_tools.py`: Contains python-callable tools mapped to the agents, including native implementations of `web_search` (using duckduckgo) and `fetch_webpage` (HTML to markdown via BeautifulSoup/Markdownify).
- **State Management**: The application handles a lot of state, including the active user context (the "room"), the transcript history, enabled agents, configuration settings, autonomous scheduling tasks, and custom personas. All mutable state (like dynamically adding models) is serialized into `data/simulator_state.json`.

### Key Technical Decisions
- **Persistent Roster Configuration**: Instead of hardcoding API keys and model specifications, `AGENT_SPECS` was factored out to `data/simulator_state.json`, allowing operators to add new aliases (e.g. `openrouter/auto`) via the `/add-model` command or the Chainlit settings UI.
- **Strict Persona Directives**: System prompts strictly require the LLMs to reply as themselves based on their trained parameter personas. The context explicitly forbids them from pretending to be multiple users or faking an IRC conversation on their own; instead, they act as single participants reacting to injected room topics.
- **Vision Integration**: Attached elements via Chainlit are caught in the `@cl.on_message` handler and mapped natively to AutoGen's `autogen_core.Image` (using PIL) and appended to the task array, allowing multi-modal models to view files directly.
- **Robust Documentation Ecosystem**: The project adheres strongly to deep, holistic documentation patterns. Root markdown files like `VISION.md`, `ROADMAP.md`, `CHANGELOG.md`, `TODO.md`, `HANDOFF.md`, and specific agent instructions (like `INSTRUCTIONS.md`, `CLAUDE.md`, `AGENTS.md`) act as global contextual anchors for future LLM iterations.
- **Synchronous vs Asynchronous Workarounds**: Due to specific Python 3.14/anyio compatibility bugs under Chainlit, the application heavily employs explicit `asyncio.to_thread` patches around sync functions like `run_sync` inside `run.py`/`app.py`. Great care was taken to ensure that async event updates (like `update_settings_ui()`) are cleanly separated from synchronous state mutations.
